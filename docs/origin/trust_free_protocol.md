# The Origin of ODD Kit: Trust-Free Protocol

> **"이걸 왜 만들었겠어. 널 못 믿어서지."**
> (2026-01-04, User)

## 1. The Core Premise (대전제)
- **Trust-Free**: 이 프로젝트는 AI 에이전트에 대한 신뢰가 없기 때문에 존재한다. 쓰레기 같이 동작하고, 병신같이 까먹고, 같은 실수를 반복하는 AI의 특성을 전제로 한다.
- **No Autonomy**: 시키는 것을 넘어서 작업을 하지 마라. 요청한 그대로 작업을 하라.

## 2. The Task Trinity (Task Lifecycle)

### 2.1. Order (명령)
- **Definition**: 작업을 시작하기 전, 에이전트가 어떻게 이해했고 어떤 동작을 할 것인지를 검사받는 문서.
- **Immutability (불변성)**: 오더를 시작하면("승인하면"), 이제 오더는 수정하지 않는다.
- **Why?**: 초기 계획(Order)을 보존해야 나중에 무엇이 왜곡되었는지 알 수 있다.

### 2.2. Progress (과정)
- **Definition**: 오더 수행 중 발생하는 모든 로그, 이슈, 버그, 수정 사항을 기록하는 문서.
- **Infinite Loop**: 사용자가 "됐다"고 할 때까지 무한히 진행하며 기록한다. 에이전트가 멋대로 완료시킬 수 없다.
- **Transparency**: 멍청한 놈이 한 작업을 사용자가 보고, 버그를 체크하고, 수정을 지시하는 모든 과정이 담겨야 한다.

### 2.3. Report (결과)
- **Definition**: 사용자가 오더를 완료시키는 시점에서, 초기 오더와 비교하여 결과적으로 무엇이 수행되었는지를 분석하는 문서.
- **The Gap**: 오더와 리포트의 차이(Gap)가 바로 "변경된 스펙"이다.

## 3. The Memory System (Atomization)

### 3.1. Task as History -> Memory
- 하나의 태스크(Order-Progress-Report)는 히스토리에 아카이빙된다.
- 이 히스토리를 분석하여 **"최종 스펙(Report)"**을 스펙 문서와 메모리 셀에 업데이트한다.

### 3.2. Granularity (버그/이슈 별 분할)
- 프로그레스의 기록을 분석하여 버그별, 이슈별로 메모리 셀을 분리한다.
- 이렇게 만들어진 수많은 메모리 셀이 다른 태스크에서 비슷하게 발생할 때 재사용되거나 합쳐진다.

### 3.3. Context Continuity
- 오더를 새로 만들 때, 단순히 오더만 생성하는 것이 아니라 기존 메모리 셀들을 참고하여 연관성을 적어둔다.
- 이를 통해 새로운 세션에서도 이전의 맥락(Context)을 이어갈 수 있다.
